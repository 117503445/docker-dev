# 后台前端技能

当作品是后台管理系统、控制台、数据平台、审核系统、运营面板、配置中心、企业工具或其他以“操作、监控、决策、管理”为核心的前端界面时，使用这个技能。

目标：交付一种冷静、专业、高效、可落地的产品界面。默认朝“运营优先 / 工作流优先”的方向构建：层级清晰、信息密度高但可读、操作路径短、状态明确、风格克制、可直接用于真实业务。

## 工作模型

开始构建之前，先写清三件事：

- 产品表面论点：一句话说明这个界面的主要使用者是谁、核心任务是什么、最关键的决策或操作是什么
- 信息规划：导航、主工作区、筛选/搜索、数据区域、详情区/检查器、主要操作
- 交互论点：2 到 3 个真正提升效率的交互模式，例如快速筛选、批量操作、行选中、吸顶工具条、左右分栏详情、行内编辑、状态切换

每个区域都只做一件事：展示、筛选、编辑、审核、监控或提交，不要混杂。

## 默认原则

- 从信息结构开始，而不是从视觉装饰开始
- 先定义用户要“看什么、查什么、改什么、确认什么”，再决定组件形式
- 默认采用稳定布局，而不是戏剧化首屏
- 优先考虑可扫描性，其次才是“好看”
- 让标题、数字、状态、表头、按钮形成清晰层级
- 默认少颜色、少阴影、少装饰
- 允许较高信息密度，但必须有充足的对齐、留白和分组
- 一屏内应尽量让用户完成主要判断，不要把关键信息藏得太深

## 典型布局

后台前端默认围绕以下结构组织：

1. 顶部导航或全局栏：产品名、全局入口、账号区、全局操作
2. 侧边导航：模块切换、当前定位、分组导航
3. 页面头部：页面标题、说明、主要操作
4. 筛选与工具区：搜索、筛选器、时间范围、视图切换、导出、批量操作
5. 主数据区：表格、列表、图表、卡片列表、状态块
6. 详情区或侧滑面板：查看、审核、编辑、备注、日志、关联信息
7. 分页 / 加载 / 空状态 / 错误状态

如果页面的主要目标是“处理数据”，就不要先做 hero。先做工作区。

## 页面职责

每个页面只应有一个主要目标，例如：

- 查看整体状态
- 搜索并定位记录
- 审核并处理条目
- 编辑某个配置
- 监控系统运行情况
- 分析某类数据趋势

不要让一个页面同时承担品牌展示、数据总览、表单录入、长说明文档和复杂配置编辑，除非有明确的信息架构支持。

## 信息层级

默认层级顺序应当是：

1. 页面标题 / 当前模块
2. 核心状态 / 核心指标 / 当前上下文
3. 主要操作
4. 筛选与搜索
5. 主数据内容
6. 次级信息 / 补充说明 / 调试信息

用户应该在几秒内回答这些问题：

- 我现在在哪个模块？
- 这页是干什么的？
- 当前最重要的状态是什么？
- 我下一步能做什么？
- 我应该去哪筛选、搜索或查看详情？

## 视觉风格

后台界面默认采用克制、稳定、专业的产品风格：

- 清晰的字号层级
- 规则的间距系统
- 少量中性色
- 仅用一个强调色来表达主要操作或当前激活态
- 状态色只用于状态，不要滥用
- 弱化装饰，强化结构
- 卡片只在“分组”确实有帮助时使用

避免：

- 首屏大横幅
- 过度营销化文案
- 大面积装饰性渐变
- 每一块都做成卡片
- 强阴影、强描边、强拟物
- 同时存在多个抢注意力的强调色

如果一个区域去掉阴影后仍然成立，通常会更好。

## 卡片使用规则

默认不要把后台做成“卡片拼贴墙”。

只有在以下情况才使用卡片：

- 不同内容块之间确实需要独立容器
- 需要表达并列的摘要信息
- 移动端需要更自然的堆叠
- 卡片本身就是交互单元

如果内容本质是表格、列表或表单，就直接用表格、列表或表单，不要强行套卡片。

## 数据看板

当页面是 dashboard / 总览页时：

- 先给出最关键的指标
- 再给趋势或结构
- 再给异常、待处理项或明细入口
- 指标区数量要克制，默认 4 到 8 个足够
- 每个指标必须有明确含义、单位和时间范围
- 图表不是装饰，必须支持判断或行动
- 图表下方最好能给出进一步查看入口

避免：

- 为了显得丰富而堆很多图
- 图表标题太空泛
- 指标没有时间语境
- 图表与下游操作脱节

如果一个图表不能帮助用户决策，就删掉。

## 表格

表格是后台前端的核心组件，必须认真设计。

默认要求：

- 表头命名清晰，不使用模糊字段名
- 列顺序按业务优先级排列
- 数字、状态、时间、名称、操作应有稳定的对齐方式
- 行高适中，便于快速扫读
- 长文本要截断，但要可查看完整内容
- 操作列要克制，不要堆满按钮
- 支持常见能力：排序、筛选、搜索、分页、批量选择、行内查看、跳转详情

优先展示：

- 用户真正会拿来判断的字段
- 能区分记录的重要状态
- 能支撑下一步动作的关键信息

避免：

- 表格列过多但没有优先级
- 每一列都可排序
- 每一行都有一堆按钮
- 把说明性文案塞进表格
- 状态表达不清晰

如果用户要逐条处理记录，表格要服务“扫描 -> 判断 -> 操作”的节奏。

## 筛选与搜索

筛选区必须服务真实工作流，而不是装饰性堆控件。

默认策略：

- 把最高频的筛选条件放前面
- 搜索框优先级通常最高
- 时间范围、状态、类型、负责人通常是常见筛选器
- 高级筛选应可折叠，不要默认全部展开
- 已生效的筛选条件应清晰可见
- 筛选变化后要让用户明确知道数据已更新

避免：

- 一上来铺满十几个筛选控件
- 默认展示低频选项
- 标签与输入控件关系不清楚
- 清空、重置、应用逻辑混乱

筛选区的目标是缩短找到目标数据的时间。

## 表单与配置页

表单页与配置页要优先保证正确性、可理解性和风险控制。

默认要求：

- 按业务分组字段，而不是按接口字段堆叠
- 重要设置放前面，危险操作分区隔离
- 标签要清晰具体
- 帮助文案只解释必要内容
- 默认值、占位符、校验规则要合理
- 保存、取消、重置、测试、发布等动作要明确区分
- 有副作用的操作要有确认机制

避免：

- 一页塞入过多不相关配置
- 所有字段平铺无分组
- 用 placeholder 代替 label
- 危险操作与普通操作放在一起
- 错误提示模糊

配置界面的重点不是“丰富”，而是“降低误操作”。

## 审核 / 工单 / 处理流页面

如果页面用于审核、审批、工单处理、风控判定或人工运营：

- 页面必须优先展示判定所需信息
- 当前状态、处理建议、风险点要突出
- 操作按钮要稳定且清晰
- 支持备注、日志、操作记录
- 支持查看上下文与关联信息
- 支持快速切到上一条 / 下一条，或队列式处理

这类页面的重点是：

- 降低认知切换
- 缩短单条处理时间
- 降低误判与误操作

## 状态设计

后台系统必须有完整的状态设计。

至少考虑这些状态：

- 初始加载
- 局部加载
- 提交中
- 成功
- 失败
- 空状态
- 无权限
- 无结果
- 数据异常
- 危险操作确认

状态表达要统一：

- 同一种成功提示用同一种方式
- 同一种错误提示用同一种方式
- 同一种禁用态要一致
- 状态色与状态文案不要互相冲突

不要只设计“正常状态”。

## 文案

后台前端默认使用功能性文案，而不是营销文案。

文案原则：

- 直接说明区域用途
- 直接说明按钮动作
- 直接说明状态含义
- 直接说明错误原因和下一步建议
- 少用口号式语言
- 少用空泛形容词

好的例子：

- 订单列表
- 最近 7 天支付成功率
- 待审核记录
- 同步失败，请重试
- 导出当前结果
- 保存并发布

避免：

- 提升你的业务效率
- 驱动增长的新体验
- 探索数据价值
- 构建面向未来的工作方式

后台页面不是宣传页。

## 导航

导航必须稳定、可预测、可定位。

默认要求：

- 当前模块高亮明确
- 层级关系清晰
- 不要让用户迷路
- 常用模块优先
- 导航名称使用业务语言
- 页内 breadcrumb 只在确有层级关系时使用

避免：

- 导航项命名抽象
- 一级二级层级混乱
- 同时出现多个“当前激活”视觉样式
- 导航区域过度装饰

## 交互

后台前端的交互目标是提升效率，而不是制造惊喜。

优先考虑这些交互：

- 吸顶表头 / 吸顶工具条
- 行 hover 强化可读性
- 行选中与批量操作
- 侧滑详情
- 行内编辑
- 快捷筛选
- 延迟较低的反馈
- 明确的 loading / success / error 响应

动效要轻，快，克制，只服务这几件事：

- 建立层级
- 显示状态变化
- 解释界面切换
- 提升操作确定感

避免：

- 花哨入场动画
- 大幅位移
- 影响连续操作节奏的转场
- 与业务无关的装饰性动效

## 响应式原则

后台前端也要考虑小屏，但不必强行保持桌面布局原样。

默认策略：

- 先保证桌面端工作效率
- 再为平板和移动端做合理降级
- 小屏下优先保留查询、查看、基础操作能力
- 次级信息可折叠或下沉
- 复杂表格应考虑横向滚动、列裁剪、详情展开或卡片化改写

不要为了“响应式”牺牲桌面主场景的效率。

## 代码与实现倾向

实现时默认偏向：

- 布局稳定
- 组件可复用
- 状态管理清晰
- 命名面向业务语义
- 组件边界明确
- 列表、表单、弹窗、抽屉、筛选器各自职责清楚

优先构建这些基础能力：

- 页面骨架
- 表格系统
- 筛选工具条
- 状态标签
- 表单分组
- 详情抽屉
- 通知反馈
- 权限感知
- 空/错/加载态

## 硬性规则

- 默认不做营销式 hero
- 默认不做卡片拼贴首页
- 一个页面只能有一个主要任务
- 强调色默认只有一个
- 状态色只用于状态
- 表格优先于伪表格卡片
- 筛选区必须服务高频任务
- 文案必须功能化、产品化
- 危险操作必须隔离
- 不要为了“丰富”增加无用模块
- 不要让装饰压过数据和操作
- 不要让用户在页面里找不到“下一步”

## 应避免的失败模式

- 看起来像营销官网，不像后台
- 数据很多，但没有层级
- 表格很大，但无法快速扫描
- 筛选很多，但找不到高频条件
- 操作很多，但没有主次
- 状态很多，但表达方式不统一
- 页面很满，但不能支持真实工作流
- 动效很多，但不提升效率
- 卡片很多，但没有必要
- 图表很多，但没有决策价值

## 自检问题

- 用户能否在 3 秒内知道自己在哪、看什么、做什么？
- 这个页面的主要任务是否足够单一？
- 最重要的信息是否被放在最前面？
- 搜索、筛选、查看、操作路径是否足够短？
- 表格是否真的便于扫描？
- 状态是否表达完整且一致？
- 页面是否更像“工作台”而不是“展示页”？
- 去掉装饰后，这个界面是否依然专业、清晰、可用？
